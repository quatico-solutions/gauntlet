# Gauntlet v1.5 — Implementation Plan

> **For agentic workers:** This plan is organized as phases with work
> packages (WPs). Cross-references to the spec (`docs/plans/2026-04-15-gauntlet-v1.5-spec.md`)
> are marked `Spec §N`. **Do not duplicate spec content here.** If a WP
> feels under-specified, read the spec section it cites; if the spec
> itself is under-specified, escalate via the amendment protocol
> (Spec §13) — do not improvise.
>
> **Phases are gates.** Phase 1 work packages may run in parallel.
> Phase 2 blocks on Phase 1. Phase 3 blocks on Phase 2 passing
> end-to-end, and is a single same-session commit.

---

## Phase dependency summary

```
Phase 1 (mechanical, parallel)
    ├── WP1.1  profiles/ → .gauntlet/context/ rename
    ├── WP1.2  browser state reset between stories
    ├── WP1.3  remove sequential-ordering assumption
    ├── WP1.4  Sonnet 4.6 floor
    ├── WP1.5  passkey decoupling from format/profile.ts
    ├── WP1.6  observer session regression verification
    └── WP1.7  generic `read` tool + tree-in-prompt  ← in Phase 1, see §1.8
         │
         ▼
Phase 2 (empirical, serial)
    ├── WP2.1  stand up two-user target
    ├── WP2.2  run Alice-and-Bob against new path
    ├── WP2.3  observe-and-describe cycle (prose-break rule)
    └── WP2.4  iteration checkpoint
         │
         ▼
Phase 3 (deletion, single commit)
    └── WP3.1  the closeout commit
```

## Phase 1 — Mechanical foundation

Phase 1 moves Gauntlet onto the new mechanical shape without touching
any agent-observable behavior that Phase 2 will measure. Every WP here
is verifiable by tests; no "run it and see" work yet.

**All Phase 1 WPs can be worked in parallel** *except* WP1.5 which
depends on WP1.1 (rename must land first so the path the passkey tool
resolves against is correct). See §1.9 for the dependency graph in
detail.

### WP1.2 — Browser state reset between stories

**Spec reference:** §5.1.

**Files touched:**

- `src/adapters/web/lib/chrome-ws-lib.js` — extend `startChrome` and
  add `clearBrowserData` helper for the remote case
- `src/adapters/web/adapter.ts` — pass `chromeProfileName` to
  `startChrome`; invoke `clearBrowserData` in remote mode; delete
  the profile dir in `close()` for local mode
- `src/cli/run.ts` and `src/api/routes/run.ts` — generate
  `chromeProfileName` per run
- `test/adapters/web/adapter.test.ts` — new coverage

**Specific changes:**

1. **Runner-side ID generation.** Add a small helper (or reuse an
   existing one — check `src/types.ts` or `src/evidence/` for a
   `makeRunId`) producing a short opaque run id (e.g., Unix ms +
   4-char random). The runner computes:
   ```ts
   const chromeProfileName = `gauntlet-run-${runId}-${card.id}`;
   ```
   Card ids are sanitized for filesystem use by Gauntlet upstream;
   any `/` or `..` in the id should already be rejected at card
   parse time. If not, sanitize here (replace non-alphanumeric with
   `-`).

2. **WebAdapter constructor.** Accept `chromeProfileName?: string`.
   If present and `!this.remote`, pass through to
   `chrome.startChrome(true, chromeProfileName)`. The existing
   `startChrome(headless, profileName)` signature already supports
   this (see `src/adapters/web/lib/chrome-ws-lib.js` around line
   1254 — do not modify the signature, just start calling it).

3. **Local cleanup in `WebAdapter.close()`.** After
   `chrome.killChrome()` (and **after** any passkey teardown and
   observer close that already run there — ordering is delicate):
   ```ts
   if (!this.remote && this.chromeProfileName) {
     const dir = chrome.getChromeProfileDir(this.chromeProfileName);
     await rm(dir, { recursive: true, force: true }).catch((err) => {
       this.logger?.logAction("chrome_profile_cleanup_failed", {
         dir, error: err instanceof Error ? err.message : String(err),
       });
     });
   }
   ```
   Expose `getChromeProfileDir` from `chrome-ws-lib.js` if not
   already exported.

4. **Remote cleanup.** Add a `clearBrowserData(tab)` to
   `chrome-ws-lib.js`. The parameter is named `tab` (not `tabIndex`)
   for consistency with its neighbors (`navigate(tab, url)`,
   `webAuthnOpenSession(tab)`, `screenshot(tab, ...)`).
   ```js
   /**
    * Best-effort reset of the current tab's browser state for the
    * remote-Chrome case where we cannot delete the --user-data-dir
    * ourselves. Cookies, cache, and the current origin's storage are
    * cleared. Silently swallows errors — a thrown error from any
    * sub-step is not fatal. The local-Chrome case uses profile-dir
    * deletion instead, which is strictly stronger.
    */
   async function clearBrowserData(tab) {
     await sendCdpCommand(tab, "Network.clearBrowserCookies", {});
     await sendCdpCommand(tab, "Network.clearBrowserCache", {});
     // Storage.clearDataForOrigin needs an origin — use the current
     // page's origin if it has one, else no-op.
     try {
       const origin = await evaluate(tab, "location.origin");
       if (origin && typeof origin === "string") {
         await sendCdpCommand(tab, "Storage.clearDataForOrigin", {
           origin,
           storageTypes: "all",
         });
       }
     } catch { /* best-effort */ }
   }
   ```
   Called from `WebAdapter.start` *after* the initial `navigate`
   but *before* the observer session opens, only when `this.remote`.

5. **Test coverage.**
   - Unit test `WebAdapter` with a mock `chrome` module verifying
     `startChrome` is called with the computed `chromeProfileName`
     in local mode.
   - Unit test that `close()` invokes the profile-dir cleanup in
     local mode and skips it in remote mode.
   - Unit test the remote mode's `clearBrowserData` invocation on
     `start()`.

**Verification criteria:**
- `bun test` green with new coverage.
- Manual smoke: run a card twice in a row with
  `--data-dir /tmp/foo`; after each run, confirm
  `ls ~/.cache/superpowers/browser-profiles/ | grep gauntlet-run`
  shows zero entries (everything was cleaned up).
- Manual smoke: run two cards concurrently (simulate via two shells)
  and confirm each gets its own profile dir, no collisions.

**Dependencies:** none. Parallel with WP1.1/WP1.3/WP1.4/WP1.5/WP1.6/WP1.7.

   - `readPasskeyFile(absolutePath: string): PasskeyCredential` —
     takes an **absolute path**. The tool resolves relative paths
     via `resolveInside` from `src/context/path.ts` (see WP1.7 step 2)

5. **Move tests.** All passkey tests live at

   `test/adapters/passkey-tool.test.ts` is **deleted**, its cases

6. **The passkey tool description must match Spec §3.4 verbatim.**

### WP1.7 — Generic `read` tool + tree-in-prompt

**Spec reference:** §3.1, §4 (especially §4.2 immutability invariant).

**Scope note.** This WP ships **one** new agent-facing tool: `read`.
No `ls`, no `grep`, no refresh path. The tree listing in the system
prompt is the agent's complete and only view of `.gauntlet/context/`
for the run, per Spec §4.2. If you find yourself wanting to add a
second tool here, stop — that path requires a spec amendment (§13).

**Files touched:**

- **New:** `src/context/read-tool.ts` —
  `buildReadTool(contextRoot): ReadTool | null`. Lives in the new
  `src/context/` top-level module (see Spec §7.1) — not under
  `src/adapters/`, because `read` is a generic filesystem primitive,
  not an adapter concern.
- **New:** `src/context/tree.ts` —
  `renderContextTree(contextRoot, options?): string` with the
  compact format from Spec §4.2. Lives in `src/context/` alongside
  `read-tool.ts`.
- **New:** `src/context/path.ts` —
  `resolveInside(root, rel): string`. The shared path guard used
  by both `src/context/read-tool.ts` and `src/adapters/web/passkey.ts`.
  Behavior matches **Spec §3.1 verbatim**, not v1's per-file path
  guards: reject `..` segments in the input, reject absolute paths,
  resolve the path against `root`, and reject any result that does
  not live under `root`. Returns the resolved absolute path on
  success; throws on failure. The v1 guards in
  `src/format/profile.ts` (lines 19–28 and 76–85) are strictly
  weaker than this — they rejected only `/` and `\` at character
  level and missed Windows absolute paths and several Unicode edge
  cases. Do not port the v1 behavior; implement the spec.
- `src/agent/prompts.ts` — `buildSystemPrompt` accepts a new
  optional `context` argument and appends the Context section
- `src/agent/agent.ts` — `runAgent` accepts the context data and
  forwards to `buildSystemPrompt`
- `src/adapters/web/adapter.ts` — wires `buildReadTool` into
  `toolDefinitions` / `executeTool`. Drops its own reference to
  `buildReadProfileTool` (but the file still exists during
  Phase 1; the *registration* is where it stops).
- `src/adapters/cli/adapter.ts` — same
- `src/adapters/tui/adapter.ts` — same
- `src/cli/run.ts`, `src/api/routes/run.ts` — compute tree, pass
  to `runAgent`
- **New tests:**
  - `test/context/read-tool.test.ts` — `read` on existing/missing
    files, error shapes. Explicit negative test: `ls`/`grep` tool
    names must not be registered. (Path guard tests live in
    `test/context/path.test.ts`.)
  - `test/context/path.test.ts` — `resolveInside` accepts valid
    relative paths, rejects `..` segments, rejects absolute paths,
    rejects escapes, handles edge cases (empty string, single-file
    root).
  - `test/context/tree.test.ts` — rendering format, byte counts,
    depth ordering, hidden file exclusion, budget truncation
    (including the new truncation-line wording from Spec §4.3)
  - `test/agent/prompts.test.ts` — Context section presence/absence,
    exact prose from Spec §4.1, including the "built once at the
    start of the run and does not change" closing paragraph
- Existing tests updated for new `runAgent` signature

**Specific changes:**

1. **`src/context/read-tool.ts` contract.**
   ```ts
   export interface ReadTool {
     definition: ToolDefinition;
     execute(args: Record<string, unknown>): Promise<ToolResult>;
   }

   export function buildReadTool(contextRoot: string): ReadTool | null {
     // Return null if contextRoot doesn't exist or is empty
     // (matches the "invisible when unused" pattern).
     // Otherwise return the ReadTool, which calls resolveInside()
     // from src/context/path.ts before reading.
   }
   ```
   Matches the shape of v1's `ProfileTool` interface for minimal
   surprise at the adapter wiring sites. The `read` tool description
   is **copy-pasted from Spec §3.1**. Do not paraphrase. If the spec
   prose contains typos, raise via the amendment protocol (Spec §13).

2. **Path guard lives at `src/context/path.ts`.** Export
   `resolveInside(root, rel): string`. **Behavior matches Spec §3.1
   verbatim**, not v1's path guards: reject `..` segments in the
   input, reject absolute paths, resolve the path against `root`
   using Node's `path.resolve`, and reject any result whose absolute
   form does not live under `root`. Return the resolved absolute
   path on success; throw on failure (the caller converts the throw
   to a tool-result error). Both `src/context/read-tool.ts` and
   `src/adapters/web/passkey.ts` import from this file. Do not
   duplicate the helper into either consumer.

3. **`src/context/tree.ts` contract.**
   ```ts
   export interface RenderContextTreeOptions {
     maxBytes?: number;      // default 64 * 1024
     maxEntries?: number;    // default 400
   }

   /**
    * Returns a rendered tree suitable for embedding in the system
    * prompt's Context section. Returns an empty string when the
    * root is absent, inaccessible, or contains no visible entries.
    */
   export function renderContextTree(
     contextRoot: string,
     options?: RenderContextTreeOptions,
   ): string;
   ```
   Format is exactly as Spec §4.2 describes. Write test fixtures
   that cover:
   - empty root → `""`
   - nonexistent root → `""`
   - single file at root
   - nested dirs
   - hidden files excluded
   - budget overflow → truncation line
   - alphabetical, case-insensitive, directories-first ordering

4. **`buildSystemPrompt` signature.** No struct. Pass the rendered
   tree string directly:
   ```ts
   export function buildSystemPrompt(
     card: StoryCard,
     contextTree?: string,
   ): string;
   ```
   When `contextTree` is undefined or empty, omit the Context section
   entirely. When present, append the exact prose from Spec §4.1 with
   `{{TREE_LISTING}}` replaced by `contextTree`. No `PromptContext`
   interface — the only data the prompt builder needs is the rendered
   string, and introducing a struct with one real field plus a
   speculative unused field is ceremony. If a future prompt-time data
   point surfaces (banner, per-run seed, etc.), that's the moment to
   introduce a struct, not now.

5. **Test Spec §4.1's exact prose.** The prompts test should
   assert the Context section matches a fixed string (loaded from
   a test fixture or inlined). This is what makes the prose
   "load-bearing": if someone quietly edits the wording, the test
   breaks and they have to either change the spec (via amendment)
   or revert.

6. **`runAgent` signature update.** Extend `AgentOptions` itself
   with `contextTree` — no intersection type, no `PromptContext`:
   ```ts
   export interface AgentOptions {
     toolTimeoutMs?: number;
     contextTree?: string;  // rendered by renderContextTree() in the runner
   }

   export async function runAgent(
     card: StoryCard,
     adapter: Adapter,
     client: LLMClient,
     logger: EvidenceLogger,
     target?: string,
     options?: AgentOptions,
   ): Promise<VetResult>;
   ```
   Forward `options?.contextTree` to `buildSystemPrompt(card, options?.contextTree)`.
   Committing `contextTree` to `AgentOptions` directly avoids the
   `AgentOptions & { context?: PromptContext }` intersection-type
   smell (intersection types in a module's public signature are a
   sign the type is under-designed).

7. **Adapters wire context tools alongside existing tools.** Each
   adapter's `toolDefinitions()` now appends `contextTools?.tools`
   at the end (after the native browser/CLI/TUI tools) and before
   the v1 `read_profile`/`install_passkey` tools if those still
   exist during Phase 1.

   **Important:** during Phase 1, both old and new tools coexist.
   `read_profile` remains registered until Phase 3. The agent will
   see both `read_profile` and `read`. This is fine — the spec's
   tool description for `read` is authoritative enough that the
   agent will prefer it, and Phase 2 is the *empirical* loop that
   confirms behavior. If Phase 2 shows the agent flails on having
   two tools to choose from, promote the Phase 3 deletion of
   `read_profile` into Phase 1.5.

8. **`executeTool` dispatch.** In each adapter:
   ```ts
   if (this.contextTools) {
     const contextToolNames = new Set(this.contextTools.tools.map(t => t.name));
     if (contextToolNames.has(name)) {
       return this.contextTools.execute(name, args);
     }
   }
   ```

9. **Runner wiring.** `src/cli/run.ts` and `src/api/routes/run.ts`
   compute the tree once before calling `runAgent`:
   ```ts
   import { renderContextTree } from "../context/tree";
   // ...
   const tree = renderContextTree(contextRoot);
   // ...
   const result = await runAgent(card, adapter, client, logger, target, {
     context: tree ? { tree, root: contextRoot } : undefined,
   });
   ```

**Verification criteria:**
- `bun test` green with all new tests.
- Manual smoke: run any existing card with a populated
  `.gauntlet/context/alice.md` and confirm the system prompt (logged
  by the action log, or dumped from a test harness) contains the
  Context section with the tree.
- Manual smoke: run with empty `.gauntlet/context/` and confirm no
  Context section appears.
- `grep -rn '{{TREE_LISTING}}' src/` returns exactly one hit (inside
  `prompts.ts`).

**Dependencies:** none directly, but it's **in Phase 1 on purpose**
(not Phase 2) — see §1.8 below for the rationale.

### 1.8 Why WP1.7 lives in Phase 1

The conclave left this as an interesting call: WP1.7 is the first WP
that changes the agent's observable surface. The prose-first discipline
of Plan E says "write the prose, lock it, then conform code to match."
Putting WP1.7 in Phase 1 looks like it jumps the gun.

**Vetinari's call: keep WP1.7 in Phase 1.**

Reasons:

1. **The prose *is* written first.** Spec §3.1 and §4.1 are
   the locked prose. WP1.7 conforms code to that prose. That is
   the Plan E discipline exactly — the sequencing is "write prose,
   ship code that matches, then observe in Phase 2." Phase 2
   discovers *new* contradictions, not initial ones.

2. **Phase 2 needs it.** The empirical loop in Phase 2 runs
   Alice-and-Bob against the *new* path. If WP1.7 is not landed,
   there is no "new path" to run against — Phase 2 would have to
   either iterate on the old path (defeating the point) or do its
   own WP1.7 (defeating the parallel-Phase-1 point).

3. **The mechanical work is mechanical.** Writing a `read` tool
   and a tree renderer and a prompt section is not *discovery*
   work. It is typing. The discovery work is "does the agent use
   it?" — that's Phase 2.

4. **The one reasonable counterargument** — "landing it in Phase 1
   means we cannot change the prose in Phase 2 without an
   amendment" — is not a counterargument, it's the *point*.
   Amendments are cheap and create a record. Silently changing
   prose on the fly is the anti-pattern.

If WP1.7 turns out to be bigger than expected during Phase 1, the
fallback is to simplify the tree format (e.g., drop byte counts, or
raise the depth at which truncation kicks in). Never punt the `read`
tool description itself, and never add a list/search tool as a
"temporary" expedient — Spec §4.2 forbids it.

### WP2.3 — Observe-and-describe cycle (the prose-break rule)

**This is the core of Phase 2.** Every failure mode observed in WP2.2
is categorized:

- **Agent's fault** (model limitation, reading comprehension, etc.) —
  no action yet; note it for the iteration checkpoint (WP2.4).
- **Prose's fault** (the tool description, system prompt section, or
  card phrasing confused the agent) — follow the **prose-break rule**:
  1. Identify the prose that misled.
  2. Draft the minimum change.
  3. Open an amendment (Spec §13) — this is a Matt-in-the-loop sync,
     should take minutes.
  4. On approval, update the spec in a single commit.
  5. Then update the code to match, in a follow-up commit.
  The sequence is critical: spec first, code second.
- **Code's fault** (the tool doesn't do what the description says;
  the tree renderer is broken; the path guard lets something
  through) — fix in code. No spec change.
- **Target's fault** (the Express app has a bug; Brainstorm is
  misconfigured) — fix the target.

**Do not conflate these categories.** A failure mode that "works
after I tweak the tool description" is *not* a code fix — it is
prose work, and if you silently changed the description you just
invalidated the spec. Reach for the amendment protocol the moment
you want to change prose.

**Keep a running log** of each failure + category + resolution in
a throwaway file (`docs/plans/2026-04-15-gauntlet-v1.5-phase2-log.md`
or similar). At Phase 2's end, this log becomes the raw material
for any amendments that landed and any observations Matt wants to
see. Do not merge the log — it's working notes.

### WP2.4 — Iteration checkpoint

**"Good enough to stop iterating" criteria.** All of the following
must be true, with evidence:

1. The Alice-and-Bob privacy card **passes end-to-end** against the
   real target (not a mock, not a truncated run, not a hand-edited
   message stream). `status: "pass"` in the result, with the
   acceptance criteria correctly judged.
2. The agent **called `read` before logging in as each identity.**
   Verified via the action log.
3. The agent **did not call any v1 tool** (`read_profile`,
   nothing else). Verified via the action log — zero
   occurrences of `read_profile`.
4. The same card runs a **second time from scratch** (fresh
   data dir, new Chrome profile) and passes again. Browser-state
   reset is not a fluke.
5. Ponder's NetworkEffect passkey smoke still passes end-to-end
   against the new path — the passkey decoupling did not regress
   the live-Chrome WebAuthn ceremony. Run the passkey-specific card
   Ponder used in v1, using `install_passkey` with its new
   `path` parameter. This is the "Phase 1 said we verified
   observer, now Phase 2 verifies passkey" check.
6. **No open prose-break amendment is in flight.** If an amendment
   is pending Matt's review, Phase 2 is not done.

When all six are true, open Phase 3.

**If the checkpoint keeps slipping** (more than two loops through
WP2.2 → WP2.3 → WP2.4 without converging), stop and escalate to
Matt. The failure mode is no longer "prose needs tuning"; it's
"the fundamental design needs re-litigation" and the conclave needs
to be reopened. Do not silently keep iterating.
