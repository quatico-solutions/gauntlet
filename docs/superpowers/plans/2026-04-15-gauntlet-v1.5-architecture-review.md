# Gauntlet v1.5 — Architecture Review

## 1. Summary

The layering in v1.5 is **broadly correct and materially better than v1**.
Vetinari's instinct to dissolve `src/format/profile.ts` is right, and the
destination of the passkey code (`src/adapters/web/passkey.ts`) is right for
exactly the reason Matt flagged: passkeys are a web-adapter concept, not a
format concept. The new `read` tool is genuinely content-neutral and
adapter-agnostic, which is what qualifies it for a shared home. There is
one layering mistake of real weight: **the home for `read` is called
`src/adapters/context-tools.ts`, sitting directly under `src/adapters/`
alongside the obsolete `profile-tool.ts` and `passkey-tool.ts`, as if it
were an adapter-level peer.** It is not. It is a filesystem tool shared by
three adapters, and it should live under `src/context/` (or, less
ideally, `src/agent/`). Promoting that one directory change upgrades the
whole refactor's layer grade from B+ to A-. Several smaller items
(PromptContext's three-hop data flow; `resolveInside`'s homelessness; a
§7.1 bullet list that contradicts itself) are cleanup-grade; see the
per-item section for triage.

---

## 2. Review scope

**Evaluated.** The layering of every new or moved file in v1.5: where it
lands, what it depends on, and whether its neighbors make sense. Vetinari's
eleven deferred layering items (prompt §11). Two additional concerns I
found that are not on his list: (a) `WebAdapterOptions.contextRoot` giving
the web adapter a filesystem responsibility it doesn't want, and (b) the
current `src/format/profile.ts` path-guard duplication versus what v1.5
plans to pull into `resolveInside`.

**Not evaluated.** The three-phase execution structure; the 10 decisions;
the tool set; the prose of the `read` / `install_passkey` descriptions;
the observer session (declared off-limits and rightly so); any question
of whether Phase 2's target app should be Express or Brainstorm. Those
are settled per Matt's directive.

**Method.** Read the spec and plan in full. Read every current file that
v1.5 touches (agent, prompts, profile, profile-tool, passkey-tool,
web/adapter, cli/adapter, tui/adapter, cli/run, api/routes/run, story-card,
evidence/logger, and the relevant slices of chrome-ws-lib.js). Cross-checked
the plan's `Files touched` lists against what the code actually exposes.

## 3. Layering map — before v1.5

The v1 code has a small, clean set of natural layers:

```
┌──────────────────────────────────────────────────────────────┐
│ runner   src/cli/run.ts                                      │  glue
│          src/api/routes/run.ts                               │  (orchestration)
├──────────────────────────────────────────────────────────────┤
│ agent    src/agent/agent.ts       ← the loop                 │  core
│          src/agent/prompts.ts     ← prompt construction      │  (transport-blind)
├──────────────────────────────────────────────────────────────┤
│ models   src/models/provider.ts, src/models/resolve.ts       │  core
│                                                              │
├──────────────────────────────────────────────────────────────┤
│ adapters src/adapters/adapter.ts        ← the interface      │  protocol-
│          src/adapters/profile-tool.ts   ← v1 keychain tool   │  specific
│          src/adapters/passkey-tool.ts   ← v1 passkey tool    │
│          src/adapters/web/adapter.ts    ← CDP wiring         │
│          src/adapters/cli/adapter.ts    ← stdio wiring       │
│          src/adapters/tui/adapter.ts    ← tmux wiring        │
├──────────────────────────────────────────────────────────────┤
│ format   src/format/story-card.ts       ← data in/out        │  pure parsers
│          src/format/profile.ts          ← ⚠ mixed            │
├──────────────────────────────────────────────────────────────┤
│ evidence src/evidence/logger.ts         ← observability      │  plumbing
│          src/evidence/writer.ts                              │
├──────────────────────────────────────────────────────────────┤
│ config   src/config.ts                                       │  bootstrap
└──────────────────────────────────────────────────────────────┘
```

**The one v1 layering violation that matters for this review** is the one
Matt flagged: `src/format/profile.ts`. Looking at the file directly
(`src/format/profile.ts`, lines 53–120), it mixes:

- `listProfiles` / `readProfile` — pure filesystem read of `<dir>/<name>.md`,
  adapter-agnostic. Plausibly belongs somewhere.
- `listPasskeyProfiles` / `readPasskey` — scans for `<dir>/<name>/passkey.json`,
  parses it as WebAuthn credential JSON, and runs `toStandardBase64` to
  patch Chrome's CDP bug.
- `PasskeyCredential` interface — the data shape WebAuthn's CDP surface
  consumes.
- `toStandardBase64` — a CDP bug-fix helper that exists solely because
  "Chrome CDP's `WebAuthn.addCredential` rejects base64url." That comment
  is in the file (lines 62–69).

Half of that is a file-format parser that would be fine in `format/`. The
other half — `PasskeyCredential`, `toStandardBase64`, and the "read from a
directory that the passkey tool knows about" shape — is a **web-adapter
implementation detail that format/ learned about because the shared code
had to live somewhere.** `format/` does not belong to a protocol; it is
supposed to be "given bytes, give me a structure." The Chrome CDP
base64url-vs-base64 bug is the single strongest tell: `format/` should
not know that Chrome exists.

That is the violation v1.5 has to heal.

The dominant layering concern is a single directory: **is there a
`src/context/` layer, or is there not?** Vetinari's spec implicitly says
no — the new code is scattered between `src/adapters/` and `src/agent/`.
My recommendation is yes, and the rest of the per-item review builds on
that.

## 5. Per-item review

### 5.1 — `src/format/profile.ts` dissolution (Spec §7.1)

**Vetinari's choice.** Delete the file outright. No successor. Passkey
code moves to `src/adapters/web/passkey.ts`; the profile-reader code
(`listProfiles`, `readProfile`) has no surviving consumer and dies with
the file.

**Alternatives considered.**

1. **Delete outright** (Vetinari's choice).
2. Leave a shim `src/format/profile.ts` re-exporting nothing, as a
   grave-marker.
3. Split into `src/format/passkey.ts` + `src/format/profile-md.ts` and
   keep both in format/.
4. Move `listProfiles`/`readProfile` (the non-passkey half) to some new
   home under `src/context/` instead of deleting.

**Analysis.** The v1 functions `listProfiles` and `readProfile` in
`src/format/profile.ts` lines 4–30 are dead once v1.5 lands — the spec's
`read` tool is strictly more general than `read_profile`, and nothing
else calls them. Alternative 4 would be keeping them as "a nicer API
for the common case of reading a `.md` file under a root," but that is
gold-plating: the spec explicitly rejects typed profile catalogs
(§3.3), so there is no reason to preserve a profile-shaped API.
Alternative 3 is worse: it leaves `format/` knowing about passkeys,
which was the original sin. Alternative 2 is pure cruft — shim files
attract callers and confuse future readers about whether the concept
is alive.

Alternative 1 is right. The `format/` directory after v1.5 contains
exactly `story-card.ts`, which is the single artifact `format/` was
ever coherent about: **markdown cards with frontmatter.** That is a
crisp definition, and it now fits.

**Recommendation.** Delete outright, as specified. **Do not** leave a
shim.

**One nit on the spec's wording.** §7.1 says "src/format/profile.ts —
*file is deleted.* No successor. Its story-card profile concept does
not survive." The phrase "story-card profile concept" is inaccurate —
story cards are in a different file entirely and are untouched. The
concept that does not survive is **"profile as a first-class typed
entity."** Consider tightening the prose.

**Classification: RIGHT. Cleanup spec nit: CLEANUP LATER.**

### 5.2 — `src/adapters/context-tools.ts` as the home for `read` (Spec §7.1 & Plan WP1.7)

**This is the one I push back on hardest.**

**Vetinari's choice.** `src/adapters/context-tools.ts` exports
`buildContextTools(contextRoot): { tools, execute } | null` and is wired
into all three adapters via their existing `toolDefinitions()` /
`executeTool()` machinery.

**What `read` actually is.** Look at what the tool does: it takes a path
relative to a filesystem root, resolves it with a standard guard against
`..`, reads the file as UTF-8, returns the contents. **It is pure
filesystem.** It has zero knowledge of a browser, zero knowledge of a
shell process, zero knowledge of tmux. It is adapter-agnostic in the
strongest possible sense: Vetinari himself notes in §7.4 that it is
"a filesystem tool, not a web tool," which is exactly right.

**What `src/adapters/` means.** Look at what currently lives there
(`src/adapters/adapter.ts`, `web/adapter.ts`, `cli/adapter.ts`,
`tui/adapter.ts`, the v1 `profile-tool.ts` and `passkey-tool.ts`). The
contract of the directory is: **"code that implements or wires into the
`Adapter` interface, which is transport-specific."** Even
`profile-tool.ts` and `passkey-tool.ts` — the v1 predecessors of
`context-tools.ts` — were not adapter-generic; `profile-tool.ts` was
wired into all three adapters because v1 made a category error (Matt's
point about passkey belonging in web), and `passkey-tool.ts` was
web-only despite its filename's silence on that.

**The problem with `src/adapters/context-tools.ts`.** Placing the
generic `read` tool under `src/adapters/` repeats the exact category
error Matt flagged. It says "this code lives under adapters because
adapters are the thing that use it," which is the same reasoning that
made `src/format/profile.ts` the wrong file for `PasskeyCredential` —
"format/ lives here because format/ is where the reader functions
went." The correct question is not "who uses the code" but "what layer
does the code *describe*." `read` describes **filesystem access scoped
to a context root**, which is a thing orthogonal to any adapter.

**Alternatives.**

1. **`src/adapters/context-tools.ts`** — Vetinari's choice. Wrong layer
   for the reasons above.
2. **`src/agent/context-tools.ts`** — colocated with prompts.ts and
   (per Vetinari) `context-tree.ts`. Defensible: the agent is the sole
   consumer, and `context-tree.ts` is already going there. But
   `src/agent/` is supposed to be the loop + prompt core that "knows
   nothing about the filesystem beyond what's handed to it" (your
   charter to me, rephrased). `read` reads files. Putting it in agent/
   blurs the line: agent/ becomes a place that does I/O.
3. **`src/context/read-tool.ts`** (+ `src/context/tree.ts`) — a new
   top-level module. The name matches the spec's noun — **"context"**
   is what v1.5 renamed the concept to, and the spec uses "context"
   hundreds of times. The module's charter would be: "everything that
   implements the semantics of `.gauntlet/context/`, agnostic of
   transport and agnostic of whether it's being consumed from the
   agent loop or a renderer or a tool." That includes the read
   primitive, the tree renderer, the path guard, and (importantly)
   the registration predicate "is there anything to register?".
4. **`src/util/context-tools.ts` or `src/util/fs.ts`** — treat it as
   utility. Worst option: nothing about the `read` tool is generic
   "utility"; it is a first-class product surface whose prose is
   load-bearing per §3.1.

**My recommendation: Alternative 3.** Create `src/context/`. It is
*the* v1.5 concept — the whole spec is about turning "profiles" into
"context" — and not having a module of that name while scattering
context-shaped code between `src/agent/` and `src/adapters/` is a
missed naming opportunity that will make the code harder to navigate
for the next Bob who walks in. A `src/context/` directory tells a
future reader: "the semantics of `.gauntlet/context/` live here —
start here when you need to understand what 'context' means in this
codebase."

The concrete shape:

```
src/context/
  read-tool.ts        ← buildReadTool(contextRoot): ReadTool | null
  tree.ts             ← renderContextTree(contextRoot, options?): string
  path.ts             ← resolveInside(root, rel): string (shared w/ passkey)
```

(The adapters still wire `buildReadTool(contextRoot)` into
`toolDefinitions()` / `executeTool()`, same as Vetinari's plan — the
only change is the import path. This is not a Theseus rewrite; it is a
`mv`.)

**Why this is worth a MUST-FIX-grade pushback rather than a nit.** The
category error Matt cited in his charter is exactly what happens when
a generic-sounding filename gets dropped into an adapter directory
because that's where its callers live. `src/adapters/context-tools.ts`
will read to the next Bob as "a kind of adapter" — the directory is
the claim — and the next time someone adds a context-adjacent helper,
the default gravity will be to add it next to `context-tools.ts`, not
to step back and ask "is this an adapter concern?" A `src/context/`
module is the corrective shape.

**Classification: SHOULD FIX during Phase 1.** It is a rename of two
files and an import-path update across five sites (each adapter + the
runner + the tests), performed as part of WP1.7 rather than as a
follow-up. Doing it now is cheap; doing it later is a re-refactor.

### 5.3 — `src/agent/context-tree.ts` as the home for `renderContextTree()`

**Vetinari's choice.** Put the tree renderer next to `prompts.ts`
because prompts consume the tree.

**Alternatives.**

1. `src/agent/context-tree.ts` — Vetinari's choice.
2. `src/context/tree.ts` — my recommendation.
3. `src/adapters/context-tools.ts` — fold it in next to `read`.
4. `src/util/tree.ts` — treat it as utility.

**Analysis.** The tree is not agent-specific: it's a rendered view of
`.gauntlet/context/` that happens to be consumed by the system prompt,
but "the thing that reads a directory and prints it compactly" is a
context concern. The tree's output format (§4.3, §4.2) is
prose that the agent reads, which is a prompt concern; but the code
that *produces* it is filesystem-reading, which is not a prompt
concern.

If Vetinari's `src/adapters/context-tools.ts` stays, then
`src/agent/context-tree.ts` is at least consistent-ish with it — both
"context things", one per consumer. But that consistency is internal
to a wrong choice. If `src/context/` exists, the tree renderer
obviously belongs there alongside `read`.

**Recommendation.** Move to `src/context/tree.ts` as part of the same
§5.2 change. If for some reason `src/context/` is rejected, leave the
tree renderer in `src/agent/context-tree.ts` — agent/ is a less-wrong
home than adapters/ would be, because the rendering is producing prose
that the agent reads, even if the reading-from-disk step is filesystem.

**One small naming nit**: the symbol `renderContextTree` is fine. The
filename `context-tree.ts` duplicates the `context-` prefix across both
files (`context-tools.ts` and `context-tree.ts`) because neither one
is inside a directory that already carries the "context" noun. Moving
them both to `src/context/` lets the filenames shrink to `read-tool.ts`
and `tree.ts`, which is crisper.

**Classification: SHOULD FIX during Phase 1**, paired with §5.2.

### 5.4 — `src/adapters/web/passkey.ts` as the new passkey home

**Vetinari's choice.** Directly under `src/adapters/web/`, at the same
depth as `adapter.ts`, not in a subdirectory.

**Alternatives.**

1. `src/adapters/web/passkey.ts` — Vetinari's choice.
2. `src/adapters/web/tools/passkey.ts` — in a subdirectory.
3. `src/adapters/web/lib/passkey.ts` — next to `chrome-ws-lib.js`.
4. Folded into `adapter.ts` itself as a private method set.

**Analysis.** The `web/` subdirectory currently has exactly two things:
`adapter.ts` (the wiring) and `lib/chrome-ws-lib.js` (the CommonJS CDP
primitives). The `lib/` directory is explicitly "the vendored / forked
CDP library" — it is not a general home for web helpers. Putting
`passkey.ts` under `lib/` would wrongly suggest it is CDP infrastructure
on par with `chrome-ws-lib.js`; it is not — it is a thin adapter **on
top of** that library, consuming `webAuthnOpenSession`.

A `tools/` subdirectory would be premature — there is exactly one tool
in this space, and creating a directory for it invites future "let's
add the other tools here" category creep that gets us back to the v1
shape where the `adapters/` root was a grab bag.

Folding it into `adapter.ts` would balloon that file past its current
~415 lines with a ~200-line passkey concern that has its own session
lifecycle (`teardown()`), its own CDP bug defense (`toStandardBase64`),
and its own credential schema. `adapter.ts` is already at the top of
its comfortable size; adding passkey would hurt.

**Recommendation.** Keep it at `src/adapters/web/passkey.ts`, exactly
as Vetinari specced. The directory-vs-subdirectory call is right: a
single `.ts` file at `web/`'s root is the simplest thing that matches
the reality ("this is web-adapter code, it is not the main adapter
file, and there is no other sibling").

**Classification: RIGHT.** No change.

### 5.5 — `src/adapters/passkey-tool.ts` deletion in Phase 1

**Vetinari's choice.** Delete outright during WP1.5. No shim, no
deprecation window.

**Alternatives.**

1. Delete in Phase 1 (Vetinari's choice).
2. Leave a re-export shim that forwards from `src/adapters/passkey-tool.ts`
   to `src/adapters/web/passkey.ts`, delete in Phase 3.
3. Keep the file and gradually move symbols out.

**Analysis.** The only callers of `src/adapters/passkey-tool.ts` are:

- `src/adapters/web/adapter.ts` (imports at line 9–13).
- `test/adapters/passkey-tool.test.ts`.

That is it. Grep in the code confirms: no API route, no CLI command,
no external consumer. A deprecation window is a tool for when you have
external callers who will notice. With two internal callers, both of
which are updated in the same WP, a shim adds confusion and zero
safety. The file goes.

One spec tension to flag: Spec §10 says `src/adapters/passkey-tool.ts`
"is deleted in Phase 3" and then immediately qualifies: "this file
only deletes at Phase 3 closeout if Phase 1 was incremental, otherwise
already gone." Plan WP1.5 says Vetinari **recommends removing outright
in Phase 1**, not leaving a shim. These two statements are consistent
in outcome but inconsistent in framing. §10's "Files deleted (from
Spec §10)" list in the plan also re-lists `passkey-tool.ts` at WP3.1
with another qualifier. The clean resolution: make §10 say "deleted in
Phase 1, see §7.1" and stop listing it in Phase 3. See §7 below for
the spec-edit recommendation.

**Recommendation.** Delete in Phase 1. Tighten §10's prose (§7 below).

**Classification: RIGHT. Spec prose nit: SHOULD FIX during Phase 1.**

### 5.6 — `test/` path choices for moved passkey code

Pick one and commit in the plan.

of `test/format/profile.test.ts`." The merged target is cleaner.

### 5.7 — WP1.7's placement in Phase 1 vs. Phase 2

**Vetinari's choice.** Keep WP1.7 (the `read` tool + tree-in-prompt) in
Phase 1.

**Vetinari's reasoning (Plan §1.8).** (1) Plan E's "prose-first" rule is
already satisfied because the spec §3.1 and §4.1 are the locked prose,
so Phase 1 is just code conforming to prose; (2) Phase 2 needs something
to observe against; (3) the mechanical work is mechanical; (4)
"amendments are cheap" is a feature, not a bug.

**Alternatives.**

1. WP1.7 in Phase 1 (Vetinari's choice).
2. Split WP1.7: the tree renderer + prompt plumbing goes into Phase 1,
   but the `read` tool itself lands in Phase 2 alongside the empirical
   loop.
3. All of WP1.7 into Phase 2.

**Analysis.** This is a layering-adjacent question (phases are
temporal layers) but it is fundamentally a sequencing call, not a
module-location call. I'll stay in my lane briefly: **the only way
WP1.7's placement touches my review is via what Phase 1 leaves
half-done if it slips.** If WP1.7 is in Phase 1 and gets partially
done, Phase 2 inherits a fractured state. If WP1.7 is in Phase 2, the
worst that happens is Phase 2 also does some mechanical work, which is
not a layering concern.

That said, Vetinari's reasoning holds up. Specifically point (2) is
load-bearing: if Phase 2 runs against the old path, Phase 2 isn't
really observing v1.5 — it is observing v1, and "the empirical loop
discovers new contradictions" stops being true. And point (1) is

### 5.8 — The `PromptContext` shape — three hops across module boundaries

**Vetinari's choice.** Runner builds the tree → runner calls `runAgent`
with an options object containing `context: PromptContext` → `runAgent`
forwards `context` to `buildSystemPrompt(card, context)`.

The shape (Plan WP1.7 step 4):

```ts
interface PromptContext {
  tree: string;    // rendered tree
  root: string;    // absolute context root
}
```

**The concern Vetinari flagged.** Three hops for what is essentially "a
string plus a path." Plausibly over-engineered.

**Alternatives.**

1. **`PromptContext` passed through three hops** (Vetinari's choice).
2. **Just pass the rendered tree string.** `buildSystemPrompt(card,
   contextTree?: string)`. `runAgent` takes an optional `contextTree`
   in its options. No `root`, no struct.
3. **Build the tree inside `runAgent`.** Runner passes `contextRoot`,
   `runAgent` calls `renderContextTree(contextRoot)` before building
   the prompt.
4. **Build the tree inside `buildSystemPrompt`.** Even further in —
   `buildSystemPrompt` takes `contextRoot` and does its own rendering.

**Analysis.**

Alternative 4 is wrong: it pulls filesystem I/O into the prompt
builder, which is a pure function today (look at
`src/agent/prompts.ts` lines 3–56: no I/O, no async, no side effects).
Keeping that purity is valuable — it's what makes the prompt
trivially testable with string fixtures. Alternative 4 breaks that.

Alternative 3 is tempting but loses the runner's control over the
tree snapshot. The spec §4.2's immutability invariant says "the tree
listing is built once at turn 0 and is authoritative for the whole
run." If `runAgent` is the builder, we have to guarantee `runAgent` is
called once per run (which it is today) — so it would work. But
moving the filesystem read into `runAgent` also blurs the agent-core
boundary in the same way Alternative 4 does, just less aggressively.

Alternative 2 is the one I'd pick. Look at what the `PromptContext`
struct actually carries:

- `tree: string` — the prose that gets interpolated into the
  system-prompt template.
- `root: string` — "absolute path to the context root, for prose."
  Vetinari himself comments: "not actually interpolated in v1.5 prose
  — see §4.1" (Plan WP1.7 step 4). In other words, **`root` is in the
  struct but unused**. Looking at the §4.1 prose, the string
  `.gauntlet/context/` appears as a literal inside the prompt; the
  runner's absolute path does not. `root` is speculative future
  API-shape that Vetinari hedged on.

So the struct is one real field and one placeholder. At that point the
struct is pure ceremony; passing the `tree` string directly is
simpler, has fewer places to go wrong, and does not promise an API
surface (`root`) that v1.5 does not deliver on. If a future use case
needs the root in the prompt, that's the moment to introduce the
struct, not now.

The three-hop concern disappears along with the struct: the runner
computes `contextTree = renderContextTree(contextRoot)` (a string),
passes it to `runAgent` as an option, and `runAgent` forwards it to
`buildSystemPrompt`. The hops are the same in count but each one is
passing a `string | undefined`, which is what the data actually is.

**Recommendation.** Simplify to Alternative 2. The signatures become:

```ts
// src/agent/prompts.ts
export function buildSystemPrompt(card: StoryCard, contextTree?: string): string;

// src/agent/agent.ts
export interface AgentOptions {
  toolTimeoutMs?: number;
  contextTree?: string;
}

export async function runAgent(
  card: StoryCard,
  adapter: Adapter,
  client: LLMClient,
  logger: EvidenceLogger,
  target?: string,
  options?: AgentOptions,
): Promise<VetResult>;
```

Two small wins: (a) `AgentOptions` stays the named options bag the
current code already has — no new options shape, no "options &
{ context }" intersection type (Plan WP1.7 step 6 proposes
`AgentOptions & { context?: PromptContext }`, which is a code smell
— intersection types in a single module's public signature mean the
type is under-designed); (b) the `options?.contextTree` goes directly
into `buildSystemPrompt(card, options?.contextTree)` — one argument,
no destructuring.

**When `PromptContext` *would* be right**: if v1.5 had multiple
pieces of prompt-time data (e.g. tree + a user-facing banner + a
per-run seed), a struct would be correct. It doesn't.

**Classification: SHOULD FIX during Phase 1** (Plan WP1.7 step 4 and
step 6 both need tightening). Low-cost and low-risk.

### 5.9 — `resolveInside(root, rel)` helper location

**Vetinari's choice.** Plan WP1.7 step 2 says "Pull it out into a small

This is a deferred decision, per Matt's charter.

3. Inside `src/adapters/context-tools.ts` as a private helper, and
   duplicate the logic into `src/adapters/web/passkey.ts`.
4. Inside `src/adapters/context-tools.ts` and **export** it to

5. Inside `src/adapters/web/passkey.ts` and have `context-tools.ts`

Option 3 is what v1 effectively does today (look at

Rejected.

Option 4 couples passkey.ts to an adapter sibling (`adapters/context-tools.ts`)

Option 1 (`src/util/path.ts`) is the boring-and-correct choice *if*

from anywhere.

pairs with §5.2: if `src/context/` exists, the helper lives next to
its two consumers (`src/context/read-tool.ts` and the import from
`src/adapters/web/passkey.ts`), and the directory reads as "the
semantics of context live here." If `src/context/` does not exist,
Option 1 is my fallback.

**Recommendation.**

- If §5.2 is accepted: create `src/context/path.ts` exporting
  `resolveInside(root, rel): string`. Both `src/context/read-tool.ts`
  and `src/adapters/web/passkey.ts` import from it.
- If §5.2 is rejected: create `src/util/path.ts` with the same export.
  **Do not** put the helper inside `src/adapters/context-tools.ts` —
  that makes passkey.ts import from a sibling adapter file, which is a
  coupling that only exists for this one function.

Either way, **do not leave it unhomed.** Plan WP1.7 step 2 as written
is a bug: "pull it out into a small helper" without picking the file
will produce two different helpers (one inline in context-tools.ts,
one inline in passkey.ts) under time pressure.

**Classification: MUST FIX before Phase 1 dispatches WP1.7.** Matt
needs to pick the file before work starts; the shape of the import
graph depends on it.

### 5.10 — `chrome-ws-lib.js` `clearBrowserData` helper signature (WP1.2)

**Vetinari's choice.** Add `async function clearBrowserData(tabIndex)`
to `src/adapters/web/lib/chrome-ws-lib.js`, invoked from
`WebAdapter.start()` in remote mode.

**Alternatives.**

1. Helper in `chrome-ws-lib.js` as Vetinari specced.

## 6. Other layering concerns (not on Vetinari's list)

### 6.1 `WebAdapterOptions.contextRoot` — the adapter learning about the filesystem

**Concern.** Spec §7.3 re-shapes `WebAdapterOptions` to accept
`contextRoot?: string`, and the constructor uses it to call
`buildInstallPasskeyTool(options.contextRoot, ...)`. The CLI and TUI
adapters grow the same option (§7.4).

That means each of the three adapter constructors now takes a
filesystem path whose only purpose is to hand off to `buildContextTools`
(or its passkey analog in the web adapter). The adapter itself does
nothing with the path — it just threads it through.

**The layering question.** Why does the adapter know about
`contextRoot` at all? In v1 the answer was "because the adapter owns
the profile tool and the profile tool needs to know where profiles
live." That answer was defensible when every adapter had its own
profile tool. In v1.5 the answer weakens: the `read` tool is
adapter-generic, and the adapter's sole job with `contextRoot` is to
hand it to a builder.

**Alternative.** The runner could build `contextTools` itself (calling
`buildContextTools(contextRoot)`) and pass the resulting tool set to
the adapter as a data structure, not a path:

```ts
// runner
const contextTools = buildContextTools(contextRoot);
const adapter = new WebAdapter({ chrome, contextTools, logger, ... });
```

The adapter then wires those tools into `toolDefinitions()` /
`executeTool()` the same way, but it never sees a filesystem path. It
owns *tools*, not *filesystem roots*. The passkey tool — which is
web-adapter-specific — is the one exception: the web adapter still
builds it (because passkey needs `tab` and `driver`, which the runner
doesn't know about), and *that* builder still takes `contextRoot`.

**Does this matter?** It's a close call. Arguments for Vetinari's
shape:

- The adapter has to build `passkeyTool` anyway (it needs
  `tab` + `driver`), so it already takes `contextRoot` for that. Adding
  `contextTools` to the same constructor call is consistent — the
  adapter is the single point of tool assembly.
- Decoupling `contextTools` from the adapter means two fields on the
  constructor instead of one (`contextTools` + `contextRoot` for
  passkey), which is slightly uglier.
- The runner already has to compute `contextRoot` for the tree
  renderer (passed into `runAgent`). Asking it to *also* build the
  tools and hand them over is "the runner now participates in tool
  assembly" — a new responsibility.

Arguments for my shape:

- The adapter today in `src/adapters/web/adapter.ts` lines 48–76 has
  a constructor that is already doing tool-building for two tools; it
  is at the limit of "how much tool assembly belongs in an adapter
  constructor." Adding a third tool source would push it over.
- Taking a filesystem path in the adapter's options has been a
  layering smell since v1: it was how passkey logic leaked into
  `format/` in the first place (format/ had to know the root, so the
  root got threaded all the way down). Shrinking `contextRoot`'s
  reach out of the adapter options removes one more place where the
  shape can re-infect.

**Recommendation.** **Leave it as Vetinari specced, for now.** The
adapter's need to build the passkey tool (which does need
`contextRoot`) is real, and two call sites for `contextRoot` (one for
`contextTools` via the runner, one for passkey via the adapter) would
be weirder than one. But **flag this as a layering tension to revisit
in a future refactor** where the adapter's constructor is simplified
further. If the passkey tool ever stops needing the adapter's
`tab`/`driver` context (e.g. if multi-tab passkey support lands, per
spec §14), that would be the moment to pull passkey-tool assembly
into the runner too and remove `contextRoot` from the adapter
options entirely.

**Classification: CLEANUP LATER.** Not a Phase 1 problem; document
the tension in the architecture review (i.e. here) and move on.

### 6.3 The v1 path-guard is duplicated three times in v1 code

**Concern.** The current code has three separate path-guard

### 6.4 `PasskeyTool` registration should *not* scan `contextRoot` for `**/passkey.json`

**Concern.** Plan WP1.5 step 1 says for the new
`buildInstallPasskeyTool`:

> Registration check: scan `contextRoot` for any `**/passkey.json`
> file. Return `null` if none. (This replaces the v1
> `listPasskeyProfiles`.)

**Analysis.** In v1, `listPasskeyProfiles` did a shallow scan — look
at `src/format/profile.ts` line 40–50: it only walks the direct
children of `dir` and checks for `<child>/passkey.json`. No recursion.
The new spec's "scan for any `**/passkey.json` file" is a **behavior
change**: a recursive glob.

Recursive scans at adapter construction time are a minor smell on
their own (the construction cost scales with context depth), but the
deeper problem is conceptual: **why is the registration predicate
"does any passkey.json exist anywhere in context?"**

Under the spec's "context is a freeform data store" principle (§2.1),
`.gauntlet/context/` is shaped however the author wants. There is no
guarantee that passkey credentials are named `passkey.json`, and
Spec §3.2 explicitly says authors pass the path as an explicit arg
to `install_passkey`. The registration predicate is orthogonal to
filename conventions.

A better registration predicate: **register `install_passkey` whenever
the web adapter is active and a non-empty `contextRoot` is available.**
The agent sees the tree, the tree shows filenames, the agent infers
which files are passkey JSON from their name and from reading
`install_passkey`'s description. If the author has no passkeys, the
agent simply does not call the tool — same as any other unused tool.

Alternatives:

1. Recursive scan for `**/passkey.json` (Plan WP1.5 as drafted).
2. Register whenever `contextRoot` is non-empty (the tree will tell
   the agent what is there).
3. Register unconditionally in the web adapter.
4. Register whenever there is at least one `.json` file anywhere in
   the tree (broader but cheaper than recursive filename match).

**Recommendation.** Option 2. The registration check is simply "does
`contextRoot` exist and contain any files?" — which is the same
predicate the `read` tool uses. Matches the spec's §2.1 principle that
the runner does not interpret filenames. Avoids the recursive glob.
Avoids the spec/code contradiction.

**Classification: MUST FIX before Phase 1 dispatches WP1.5.** It's a
small tweak to the plan (WP1.5 step 1) and a real layering concern
(the filename convention has leaked from the author's README into the
adapter's registration check). Matt should confirm.

### 6.5 `AgentOptions` as a forward extension point

**Concern.** Plan WP1.7 step 6 proposes `AgentOptions &
{ context?: PromptContext }` for `runAgent`'s options argument. The

adding a thing that doesn't quite fit into the existing shape." That

```ts
export interface AgentOptions {
  toolTimeoutMs?: number;
  contextTree?: string;  // per §5.8; rename from "context" for clarity
```

```ts
export async function runAgent(
```

because they encode "I didn't want to commit to extending the base

honest, and type-checkable.

**Classification: SHOULD FIX during Phase 1.** Pairs with §5.8.

## 10. Closing

The refactor is in good shape. The layering instincts that landed the
passkey code in `src/adapters/web/passkey.ts` are correct and
precisely answer the category error Matt flagged. What's missing is
the acknowledgment that v1.5 invents a new **concept** — "context" —
and therefore deserves a **module** of that name. Scattering
context-shaped code between `src/adapters/` and `src/agent/` is a
holdover from the v1 directory shape, not a reflection of what v1.5
is actually about. Creating `src/context/` is the last mile of the
same move that dissolves `src/format/profile.ts`: both ask the
question "where does this code live in the floor plan?", and both
answer "in the layer that describes what it is, not in the layer
that currently happens to call it."
